## Problem Statement

완전 오프라인(에어갭) Windows 11 사내 환경에서, 성능이 높지 않은 PC에서도 돌아가는 **로컬 LLM(gemma4:e4b-it-Q4)** 기반으로:

- 사용자가 자연어로 “이 작업을 해줘(ANSYS APDL / pyansys / SpaceClaim / Workbench)”라고 말하면
  1) **오류 가능성이 낮은 코드(초안)**를 만들고
  2) 사용자가 붙여넣은 에러 로그를 바탕으로 **수정 코드를 생성**하는
  코드 생성/수정 보조를 만들고 싶다.

현재 가장 큰 문제는 **문법/사용법 오류(특히 pyansys API 호출, 인자/순서/객체 라이프사이클)**가 종종 발생하는 것.
매뉴얼이 방대해서 “그냥 읽고 코딩”을 모델에게 시키면 실패한다.
또한 SpaceClaim/Workbench를 외부에서 직접 조작하는 자동화는 난이도가 높아, 1차 목표는 “코드 생성/수정”에 둔다.

## Primary Use Cases (대표 산출물)

1. **SpaceClaim 영역 특성 추출**
   - 원하는 영역의 x/y/z 방향 질량, 관성모멘트, 질량중심 위치
   - 영역의 대략적인 형상 특징 파악(예: 실린더면 내경/외경, 원판이면 외경/두께 등)
   - (추가) 좌표 범위를 주면 해당 구간으로 “잘라내서 선택”하고, 구간이 여러 개면 segment별로 각각 산출
2. **Workbench 해석 결과 후처리(pyansys)**
   - Workbench 결과를 pyansys로 가져오기
   - 변위(속도/가속도), 반력, 응력 후처리
   - 응력 선형화(stress linearization)
3. **솔리드 → 빔/쉘 모델 변환 및 튜닝**
   - 1)에서 얻은 형상/질량/관성 정보를 이용해 빔/쉘 모델 생성
   - 솔리드 모델과 유사한 모드해석 결과를 갖도록 튜닝 반복(파라미터 조정)

## What Makes This Cool

- 에어갭 환경에서도 “코드 생성 → (에러 로그 기반) 자기수정”이 돌아가면, 반복 작업과 디버깅 시간이 크게 줄어든다.
- 모델이 매뉴얼 전체를 읽지 않아도, **필요한 부분만 근거로 끌고 와서** 코드를 쓰게 만들면 작은 모델에서도 신뢰도가 올라간다.

## Constraints

- **에어갭**: 런타임/의존성/모델/문서/인덱스는 모두 사전 반입이 필요.
- **모델 제약**: gemma4:e4b-it-Q4가 한계. 긴 컨텍스트/복잡 추론은 제한적.
- **매뉴얼 두꺼움**: “문서 전체를 프롬프트로”는 불가. 반드시 retrieval(검색/발췌) 중심이어야 함.
- **문서 포맷 이질적**:
  - ANSYS 매뉴얼: OCR된 PDF
  - SpaceClaim 도움말: CHM
  - pyansys 레퍼런스: 온라인에 있어 오프라인 반입 필요
- **실행 환경**: Windows 11 + 사내 보안 정책(설치/실행 권한, 네트워크 차단).
- **도구 목표**: VSCode의 Cline/Continue 같은 “대화형 에이전트” UX를 유지하고 싶다.
- **스코프 조정**: 1차 목표는 “오류 없는 코드 생성(ANSYS APDL + pyansys)”과 “에러 로그 기반 수정”이다. 실행 자동화는 보류한다.

## Premises

1. 작은 로컬 모델도 **정확한 문서 발췌 + 강제 검증 루프**가 있으면 오류율을 유의미하게 낮출 수 있다 — agree/disagree?
2. “자율 에이전트”의 핵심은 LLM이 아니라, **(a) 문서 retrieval (b) 실행/테스트 (c) 실패 시 수정 규칙**이다 — agree/disagree?
3. 매뉴얼의 가치 있는 부분은 “레퍼런스 전체”가 아니라, 실제 코딩에 쓰이는 **API 호출 패턴/예제/주의사항**으로 압축 가능하다 — agree/disagree?

## Common Failure Modes (현재 관측된 오류 패턴)

- **재료 물성(포아송비) 키워드 혼동**
  - 기대: APDL에서 포아송비는 보통 `PRXY`(예: `MP,PRXY,mat,nu`)
  - 실제: 모델이 `nu` 같은 이름을 그대로 “키”로 써서 오류가 나거나, 환경에 맞지 않는 표기를 생성
- **해석 타입 혼동**
  - 요구: 모드해석(modal)
  - 실제: static 해석으로 흐름이 바뀌거나 설정이 섞임

## Approaches Considered

### Approach A: 최소 MVP — “문서 기반 스니펫 + 실행-수정 루프” (Minimal viable)

**Summary**: 매뉴얼을 전부 위키화하지 않고, “질문에 필요한 섹션”만 찾아서 인용한 뒤 코드를 생성한다. 실행 자동화 대신, 사용자가 붙여넣은 에러 로그를 기준으로 관련 섹션을 재검색하고 수정 패치를 만든다.

**Effort**: M  
**Risk**: Med

**Pros**
- 에어갭/저사양에서 현실적으로 돌아갈 가능성이 가장 높다.
- “문서 발췌 → 코드”를 강제해서 환각/추측을 줄인다.

**Cons**
- 문서 인덱싱 품질이 낮으면 모델이 잘못된 섹션을 가져올 수 있다.
- 실행 결과를 시스템이 직접 관측하지 못하므로, 사용자가 에러 로그를 제공해야 한다.

**Key design choices**
- Retrieval은 “임베딩 + 키워드(BM25)” 하이브리드(작은 모델 보완).
- 프롬프트에 **발췌된 문서 조각만** 넣고, “출처 없는 API 호출 금지” 규칙을 둔다.
- 수정 루프의 진실은 “에러 로그 + 근거 조각 교체”다. LLM 설명은 부차적.

**Guardrails (룰 기반 최소 안전장치)**
- APDL “금지/강제” 체크:
  - 사용자가 “포아송비/nu”를 말해도 출력 코드는 `PRXY`로 표준화하고, 근거 문서 조각에 `PRXY`가 없으면 생성 금지
  - 사용자가 “모드해석”을 말하면 `ANTYPE,MODAL`(또는 환경 표준) 같은 “모달 고정 토큰”이 포함되어야 함. `STATIC` 관련 토큰이 있으면 경고/재작성
- 결과물에는 항상 “근거 ID(문서/페이지/토픽)”를 남긴다(사람이 추적/감사 가능).

### Approach B: 이상형 — “오프라인 Agent Workbench(로컬 RAG + 도구호출 + 상태머신)” (Ideal architecture)

**Summary**: Continue/Cline이 호출할 수 있는 로컬 백엔드(예: 로컬 HTTP/MCP)를 둔다. 백엔드는 매뉴얼 인덱스/소스 관리/실행 샌드박스를 갖고, 에이전트는 상태머신으로 움직인다(계획 → 작성 → 실행 → 디버그 → 검증).

**Effort**: L  
**Risk**: High

**Pros**
- “지식(매뉴얼) + 실행(검증)”이 분리돼, 모델이 작아도 시스템이 견고해진다.
- 사내 팀이 쓰기 쉬운 형태(버전 관리, 재현 가능, 감사 가능)로 만들 수 있다.

**Cons**
- 에어갭 반입/업데이트 부담이 크다(모델/인덱스/런타임).
- VSCode 플러그인/에이전트와의 연결(설정/권한)이 복잡해질 수 있다.

**Key design choices**
- 도구 호출 인터페이스: MCP(가능하면) 또는 로컬 REST.
- 작업 기록: “어떤 문서 근거로 어떤 코드를 썼고 어떤 실행을 했는지” 로그 저장.
- 안전장치: 파일/명령 실행 allowlist, 프로젝트 디렉토리 샌드박스.

---

### Approach C: 창의적 — “레시피 DSL(작업 템플릿) + 코드 생성” (Creative/lateral)

**Summary**: 자주 하는 작업을 “레시피”로 표준화한다. LLM은 파이썬 코드를 직접 쓰기보다, 먼저 DSL/JSON 스펙을 채운다. 그 다음 규칙 기반 템플릿이 코드를 생성한다. LLM은 스펙/파라미터 보정에 집중한다.

**Effort**: M~L  
**Risk**: Med

**Pros**
- 문법/사용법 오류를 구조적으로 줄이기 좋다(템플릿이 안전한 패턴을 강제).
- 작은 모델에도 매우 유리하다(생성 공간을 좁힘).

**Cons**
- DSL 설계가 잘못되면 유연성을 잃는다.
- “비정형 작업”에선 DSL이 오히려 장애물이 될 수 있다.

## Recommended Approach

**Recommendation: Approach A**  
이유: 지금 제약(에어갭 + 저사양 + 작은 모델 + 문서 방대)에서 “바로 가치”를 내는 최단 경로다. 핵심은 위키가 아니라 **(1) 문서 발췌 품질 (2) 출처 강제 (3) 에러 수정 루프의 규율**이다.

Approach A로 “문법/사용법 오류율이 눈에 띄게 감소”하는 걸 증명하면, 그 다음에 Approach B(워크벤치화)나 C(레시피 DSL)로 확장한다.

## DPF Postprocess Template (Workbench 결과 후처리 방향)

DPF로 갈 때 “작은 모델 + 문서 두꺼움”에서 안정성을 올리는 핵심은:

- 코드 스켈레톤(정석 패턴)을 템플릿으로 고정
- 결과 연산자는 **DPF operator 이름을 근거로**만 호출(출처 없는 호출 금지)
- 출력(그래프/테이블/CSV)을 표준화해 사람이 검증하기 쉽게 만들기

### 최소 템플릿(골격)

- `dpf.Model(<rst_path>)`로 모델 생성
- `model.results`에서 필요한 결과를 요청
  - 변위: displacement
  - 반력/반력계: reaction force 계열
  - 응력: stress 계열
- 시간/주파수(특히 모드해석)는 time/frequency scoping을 명시

### 응력선형화(stress linearization)

DPF에 “워크벤치 UI의 응력선형화”가 그대로 있는지(또는 버전별 operator 지원)와 무관하게,
1차 PoC에서는 다음 2단계로 나누는 게 안전하다:

1) **선형화에 필요한 입력을 표준 출력으로 확보**
   - SCL(Stress Classification Line) 정의에 필요한 노드/요소 경로 추출(수동 또는 간단한 스크립트 입력)
   - 경로 위의 응력 텐서 샘플을 일정 개수로 뽑아 CSV로 저장
2) **선형화 계산은 별도 함수로 구현**
   - 막 두께 방향 좌표계/성분(hoop/meridional 등) 정의가 필요할 수 있으니 “계산 가정”을 명시
   - 최소 목표는 “막력막힘(막 + 굽힘 분해)”을 반복 가능하게 만드는 것

이렇게 쪼개면, 도구/버전 변화가 있어도 “데이터 추출”만 안정적으로 되면 나머지는 점진적으로 맞출 수 있다.

## Document Ingest Plan (오프라인 문서 반입/인덱싱)

핵심은 “위키화”보다 **검색 가능한 근거 조각을 안정적으로 뽑아오는 것**이다.

### ANSYS PDF (OCR)

- 목표: 페이지/섹션 단위로 텍스트를 추출하고, “문서명 + 섹션 + 페이지” 메타데이터를 유지한다.
- Retrieval 기본은 키워드(BM25/FTS)로 시작하고, 필요하면 작은 임베딩 모델을 추가한다.

### SpaceClaim CHM

- 목표: CHM을 HTML/텍스트로 풀어낸 뒤, 제목/토픽 경로(목차)를 메타로 유지한다.
- 오프라인에서 가능한 전략:
  - (실무) CHM decompile로 HTML 추출 후 인덱싱
  - (차선) CHM을 “토픽별 txt”로 평탄화해서 검색 정확도를 높임

### pyansys 문서 (온라인 → 오프라인 반입)

우선순위는 다음 순서가 안전하다:

1) **패키지 자체에서 얻는 로컬 레퍼런스**
   - 오프라인 환경에서 설치되는 pyansys 버전에 대해, docstring / 타입힌트 / `help()` 기반 레퍼런스를 1차 근거로 사용
   - 장점: “내가 실행하는 버전”과 문서가 일치할 확률이 높다
2) **HTML 문서 미러링**
   - 온라인 가능한 준비 머신에서 pyansys 공식 문서(HTML)를 통째로 받아서 오프라인으로 반입
   - 장점: 예제/가이드가 풍부
   - 단점: 반입/업데이트 오퍼레이션 필요

## Retrieval Rules (작은 모델을 살리는 규율)

- “근거 없는 호출 금지”: 검색 결과(매뉴얼 발췌/로컬 docstring)에서 **근거 조각이 확보된 API만** 호출.
- “Signature 우선”: 함수/메서드 시그니처(인자/타입/기본값)를 근거 조각에서 먼저 확인하고 코드 생성.
- “에러 기반 재검색”: 실패하면 (에러 메시지 키워드 + 관련 심볼)로 재검색하고, 근거 조각을 교체한 뒤 패치.

## Success Criteria

- (정량) 상위 N개 대표 작업(예: 10개)에 대해 “초안 코드가 실행 가능한 수준” 70%+ (사람이 작은 수정 없이 돌아가는 비율).
- (정량) 대표 오류 3종에 대해 “에러 로그 입력 시 1~2회 수정 제안 내 해결” 60%+.
- (정성) 생성 코드에 “근거 섹션/예제 링크(로컬 경로)”가 남아, 사람이 추적 가능.
