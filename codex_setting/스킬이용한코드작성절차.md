# 스킬 이용한 코드 작성 절차 (gstack + Karpathy)

이 문서는 Codex에서 **gstack 워크플로우 스킬**과 **Karpathy(karpathy-guidelines) 규율 스킬**을 같이 써서, 계획→구현→검증→출시까지 흔들리지 않게 진행하는 “기본 루틴”이다.

---

## 0) 기본 개념 (역할 분리)

- **gstack**: “무슨 순서로, 어떤 산출물을 만들고, 어떤 게이트를 통과할지”를 제공하는 **프로세스/체크리스트**
- **Karpathy(karpathy-guidelines)**: 과도한 변경/스코프 크립/검증 부재를 줄이는 **행동 규율**

요약하면: **gstack = 흐름**, **Karpathy = 태도(제약)**.

---

## 1) 추천 절차 (가장 흔한 루트)

1. **문제 정의**: `/office-hours`
   - 사용자 문제, 범위, 성공 기준(“이러면 끝”)을 문장으로 못 박기
2. **계획 수립/고정**: `/autoplan`
   - 작업을 작게 쪼개고, 각 단계의 “검증 방법(테스트/커맨드/화면 확인)”을 포함시키기
3. **(필요 시) 계획 강화**: `/plan-eng-review` (또는 `/plan-design-review`, `/plan-ceo-review`)
   - 리스크/엣지케이스/테스트 매트릭스/롤백 전략까지 보강
4. **구현 모드 진입(규율 ON)**: `/karpathy-guidelines`
   - *최소 변경*, *가정 명시*, *검증 가능 목표* 원칙을 활성화
5. **코드 작성(구현)**: 일반 프롬프트로 진행
   - “계획의 1단계만 구현해줘”처럼 **단계별로** 구현/검증 반복
6. **리뷰 게이트**: `/review`
   - 불필요한 리팩토링/성능·보안·에러처리 누락/회귀 위험 점검
7. **QA 게이트**: `/qa` (리포트만이면 `/qa-only`)
   - 사용자 흐름 기반으로 실제로 눌러보고, 발견된 버그를 고치고 재검증
8. **최종 체크/리포트**: `/ship`
   - 마지막 확인(테스트/체인지로그/릴리즈 노트 등) + 요약 리포트

---

## 2) “코드는 언제 쓰나?”

- **코드 작성은 “계획을 고정한 다음”**에 시작한다.
- 구현은 한 번에 크게 밀지 말고, **autoplan의 단계(마일스톤) 단위로**:
  - (예) 1단계 구현 → 로컬 테스트/빌드 → 2단계 구현 → 테스트 …
- 막히면 다시 `/plan-eng-review`로 “막힌 이유(가정/리스크/테스트)”를 정리하고, 계획을 조정한 뒤 재개한다.

---

## 3) 품질을 올리는 운영 팁 (짧게)

- **Karpathy는 “구현 직전 + /review 직전”**에 한 번씩 켜면 효과가 가장 크다.
- 성공 기준은 반드시 “검증 가능한 형태”로:
  - “동작하게” ❌ → “`npm test` 통과 + 3개 케이스 추가 + 버튼 클릭 시 toast 노출” ✅
- 의도치 않은 변경을 줄이려면:
  - “이번 PR에서 건드리는 파일 목록”을 먼저 제한하고, 그 밖은 “발견만 하고 언급”으로 끝내기

---

## 4) 짧은 예시 템플릿 (그대로 복붙)

1) `/office-hours` → 요구/성공기준 확정  
2) `/autoplan` → “단계별 검증 방법” 포함해서 계획 고정  
3) (복잡하면) `/plan-eng-review`  
4) `/karpathy-guidelines`  
5) “계획 1단계만 구현해줘. 검증은 X 커맨드로.”  
6) `/review` → 수정 반영  
7) `/qa`  
8) `/ship`
