## Goal (PoC v1)

에어갭 Windows 11에서 로컬 LLM(`gemma4:e4b-it-Q4`)을 사용해:

1) ANSYS APDL / pyansys(= DPF) 코드 초안을 “근거(매뉴얼 발췌)” 기반으로 생성  
2) 사용자가 붙여넣은 에러 로그를 입력으로 받아 “근거 재검색 → 최소 패치”로 수정 코드를 제안

실행 자동화(Workbench/SpaceClaim을 띄워 돌리기)는 v1 범위에서 제외.

## Non-goals (v1)

- Workbench/SpaceClaim GUI 자동 조작(헤드리스/라이선스 포함)
- “정답 검증”까지의 완전 자동 루프(테스트/수치 검증)
- 매뉴얼 전체 위키화(대규모 문서 재작성)

## Deliverables

- 로컬 문서 인덱스(검색 가능):

  - SpaceClaim CHM(HTML/TXT로 추출 후)
  - pyansys: 1차로 로컬 docstring/type hints, 2차로 HTML 미러(선택)
- VSCode 에이전트 연결:

- 템플릿/가드레일:
  - APDL: `PRXY` 고정, modal/static 혼동 방지
  - DPF: 모달 `--mode` 강제, 결과 builder 디스커버리 우선
- 운영 문서:

  - 사용 프롬프트 템플릿(요청/에러수정)
  - SpaceClaim 추출값 → 빔/쉘 모델 입력으로 넘기는 CSV 스키마
  - 모달 튜닝 목표(주파수/유효질량) CSV 스키마

## Beam/Shell Model Tuning (추가 범위)

요구사항:
- SpaceClaim 추출 결과를 기반으로 “빔/쉘 모델 파라미터”를 CSV로 정규화
- 사용자가 부족한 정보를 같은 스키마로 쉽게 채울 수 있어야 함
- 튜닝 범위를 min/max로 제공(예: 내경/외경/두께/길이 범위)
- 목표는 “모드번호 기준” 주파수 + 방향별 유효질량(필수)
- 고급: 솔리드 모델 모드형상과 MAC 비교(옵션)
- 튜닝 이후에도 연계용 노드를 “리튠 없이” 추가할 수 있어야 함(노드 번호 충돌 회피)

## Architecture (최소 구성)

**A) 로컬 문서 인덱스**
- 저장소: SQLite(FTS5) + 원문 조각 파일(또는 SQLite blob)
- 단위: “근거 조각(chunk)” = {source_id, title, page/topic, text, path, locator}
- 검색: 키워드 중심(FTS) + (선택) 임베딩 추가

**B) Retrieval + Prompt Assembler**
- 입력: 사용자의 요청(자연어) + (선택) 코드 스켈레톤/컨텍스트
- 출력: “근거 조각 3~8개” + “생성 규율(금지/강제)” + “코드 템플릿”

**C) Code Generator / Patch Generator**
- 생성 모드:
  - `generate`: 근거 조각 기반 코드 생성(APDL / Python DPF)
  - `fix`: 에러 로그 + 실패 코드 + 근거 ID → 재검색 → 최소 패치
- 가드레일:
  - 룰 기반 검사(정규식/토큰): `PRXY` 강제, `ANTYPE,MODAL` 강제 등
  - 위반 시 “재검색 요청”으로 전환(지어내기 금지)

**D) VSCode Integration**
- Continue/Cline은 “대화 UI”로 사용
- 오프라인에서 가능한 형태로:
  - 모델 호출: 로컬 LLM 서버(Ollama 등) + 로컬 엔드포인트
  - 문서 검색: (1) 로컬 스크립트 실행, 또는 (2) 로컬 HTTP/MCP 서비스(선택)

## Data Model (문서 조각)

`doc_chunks` (SQLite):
- `id` (pk)
- `source` (ansys_pdf | spaceclaim_chm | pyansys_docs | pyansys_docstring)
- `title`
- `locator` (예: "p.123", "Topic/Path", "module:function")
- `path` (로컬 파일 경로)
- `text`
- `tags` (csv 또는 json)

## Ingest Plan

### 1) ANSYS PDF (OCR)

- 변환: PDF → 텍스트(+페이지 번호 유지)
- chunking: 페이지 단위 또는 “헤더/섹션” 기준(가능하면)으로 300~1200 토큰 범위
- 메타: `docname`, `page`, `section`(있으면), `pdf_path`

### 2) SpaceClaim CHM

- 변환: CHM → HTML/TXT로 추출
- chunking: 토픽 단위(제목/목차 경로 유지)로 300~1200 토큰
- 메타: `topic_path`, `topic_title`, `html_path`

### 3) pyansys

**1차(권장)**: 오프라인에 설치된 패키지에서 docstring/type hints 추출  
**2차(선택)**: 온라인 준비 PC에서 HTML 문서 미러링 후 오프라인 반입

## Prompt Templates (핵심 규율 포함)

### Generate (APDL)

- Inputs: 목적, 모델/요소/재료 파라미터, 해석(modal/static), 출력 요구
- Rules:
  - “포아송비는 `PRXY`로만 표현”
  - “모드해석이면 모달 고정 토큰 포함, static 토큰 금지”
  - “근거 조각에 등장하지 않는 키워드/명령은 사용 금지(필요 시 재검색)”

### Generate (DPF, *.rst)

- Inputs: `rst_path`, `analysis=modal`, `mode=1..N`, `results=[disp,stress,rf]`, scoping
- Rules:
  - “모달은 모드번호로만(주파수 값 사용 금지)”
  - “가능한 result builder 목록을 먼저 확인하고, 없는 builder는 호출 금지”

### Fix (에러 로그)

- Inputs: 에러 로그, 실패 코드, 근거 ID
- Steps:
  1) 에러 키워드/심볼 추출
  2) 근거 재검색(기존 근거 ID 우선 + 에러 키워드 확장)
  3) 최소 패치 생성
  4) 패치 이유를 “근거 조각”으로만 설명(추측 금지)

## Implementation Steps (2주 PoC)

### Week 1: 문서 인덱스 + DPF/APDL 템플릿 고정

1. Ingest 스크립트 설계(파일 트리 + SQLite 스키마)
   - `offline_agent/index/build_index.py`로 `ansys/ spaceclaim/ pyansys/` 폴더를 통째로 인덱싱
   - 파일명은 고정하지 않고 “폴더 기준”으로 수집
2. ANSYS PDF 텍스트 추출 파이프라인 작성/검증(페이지 메타 유지)
   - 에어갭에서 poppler 설치가 어려우면 `<pdf>.txt` 사전 추출 방식도 허용(동일 basename)
3. CHM 추출 파이프라인 확정(Windows 커맨드 기반) + 샘플 1개로 검증
   - 기본은 `hh.exe -decompile` 사용(Windows 내장), 실패 시 권한/정책 확인
4. 검색 API(로컬 CLI):
   - `search "<query>" --source ansys_pdf --top 8`
   - 구현: `offline_agent/index/search.py`
5. 가드레일 룰셋 v1:
   - `PRXY` 강제
   - modal/static 혼동 방지
6. DPF 템플릿을 “모달 모드번호” 중심으로 고정
   - 참고: `/Users/yonghwaheo/Documents/antigravity/codex_setting/dpf_postprocess_template.py`

### Week 2: VSCode 연동 + 에러수정 루프 PoC

1. Continue/Cline에서:
   - “문서 검색”을 호출하는 방법(스크립트 실행 or 로컬 서비스) 선택
2. Generate 프롬프트 템플릿을 3개 유스케이스에 맞게 확정
   - SpaceClaim 영역 특성 추출
   - DPF 후처리(모달 + stress linearization 준비)
   - APDL 기반 빔/쉘 모델링(모달)
3. Fix 프롬프트 템플릿 + 최소 패치 룰 적용
4. 측정:
   - 대표 작업 10개에 대해 “초안이 바로 실행 가능” 비율
   - 에러 로그 10개에 대해 “1~2회 수정으로 해결” 비율

## Acceptance Criteria (PoC 종료 조건)

- 문서 검색이 “페이지/토픽/경로”까지 포함해 안정적으로 근거를 뽑는다.
- APDL에서 `PRXY`/모달 강제 룰이 실제로 혼동을 줄인다.
- DPF는 `--mode` 기반으로 결과를 뽑아 CSV를 만든다(결과 builder는 디스커버리 기반).
- 사용자가 에러 로그를 넣으면 “근거 재검색 → 최소 패치” 흐름이 유지된다(추측 수정이 아니라).

## Risks & Mitigations

- DPF API/버전 차이로 연산자 이름이 다름
  - Mitigation: 디스커버리 출력 → 버전별 매핑 테이블을 따로 관리
- 문서 chunk가 너무 크거나 작아 retrieval 품질이 흔들림
  - Mitigation: chunk size를 2~3개 프리셋으로 실험(섹션/페이지 기준)
- 작은 모델의 장문 추론 한계
  - Mitigation: 템플릿+가드레일로 생성 공간을 좁히고, 근거 조각을 3~8개로 제한
